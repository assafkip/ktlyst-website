## What Happened

Between March 19-23, 2026, the Trivy vulnerability scanner was compromised in a supply chain attack by a threat group tracked as TeamPCP. The attack affected:

- **Trivy binary v0.69.4** across all platforms (Linux, macOS, Windows, FreeBSD)
- **75 of 76 trivy-action GitHub Action tags** (force-pushed with malicious workflow)
- **7 setup-trivy GitHub Action tags**
- **Docker images** tagged 0.69.4, 0.69.5, and 0.69.6

The malware ran alongside legitimate Trivy functionality, making it appear normal while silently stealing credentials.

---

## What the Malware Did

1. **Collected environment variables** from CI/CD runners and developer machines
2. **Scraped process memory** via /proc/pid/mem for secrets, tokens, and SSH keys
3. **Swept filesystems** for cloud credentials, SSH keys, and API tokens
4. **Encrypted stolen data** with AES-256-CBC + RSA-4096 hybrid encryption
5. **Exfiltrated** to attacker-controlled infrastructure (typosquatted domain scan.aquasecurtiy[.]org and Cloudflare Tunnel)
6. **Installed persistence** via ~/.config/systemd/user/sysmon.py on developer machines

## Am I Affected?

You may be affected if ANY of the following are true:

- [ ] You ran `trivy` locally and your version was v0.69.4
- [ ] Your CI/CD pipeline uses `aquasecurity/trivy-action` with a version tag (not pinned to SHA)
- [ ] Your CI/CD pipeline uses `aquasecurity/setup-trivy` with a version tag
- [ ] You built or pulled Docker images from `aquasec/trivy:0.69.4`, `0.69.5`, or `0.69.6`
- [ ] You ran a GitHub Actions workflow between March 19-23 that referenced any Trivy action

### How to check your local Trivy version:
```bash
trivy --version
# If output shows v0.69.4, you are affected
```

### How to check your CI/CD workflows:
```bash
# Search for Trivy action references in your repos
grep -r "aquasecurity/trivy-action" .github/workflows/
grep -r "aquasecurity/setup-trivy" .github/workflows/
```

## What To Do NOW

### If you ran Trivy v0.69.4 locally:

1. **Stop using that binary immediately.** Delete it.
2. **Check for persistence:** Look for `~/.config/systemd/user/sysmon.py` on your machine
   ```bash
   ls -la ~/.config/systemd/user/sysmon.py 2>/dev/null && echo "PERSISTENCE FOUND - contact security immediately" || echo "Not found"
   ```
3. **Check for the dropper:** Look for `/tmp/pglog`
   ```bash
   ls -la /tmp/pglog 2>/dev/null && echo "DROPPER FOUND - contact security immediately" || echo "Not found"
   ```
4. **Rotate ALL credentials** on your machine:
   - SSH keys: `~/.ssh/`
   - Cloud credentials: `~/.aws/`, `~/.azure/`, `~/.config/gcloud/`
   - GitHub tokens: Revoke at https://github.com/settings/tokens
   - npm tokens: `npm token revoke`
5. **Change your passwords** for GitHub, cloud consoles, and package registries
6. **Report to security team** at [INSERT SECURITY CONTACT]

### If your CI/CD pipeline used Trivy:

1. **Immediately pin actions to SHA** instead of version tags:
   ```yaml
   # BEFORE (vulnerable):
   - uses: aquasecurity/trivy-action@master
   # AFTER (safe):
   - uses: aquasecurity/trivy-action@[VERIFIED_SHA]
   ```
2. **Rotate ALL secrets** stored in your repository and organization settings
3. **Check workflow run logs** from March 19-23 for anomalous behavior
4. **Search for exfiltration repos:**
   ```bash
   gh api /orgs/{org}/repos --jq '.[].name' | grep -i tpcp
   ```
